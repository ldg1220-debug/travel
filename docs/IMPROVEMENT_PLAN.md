## 1. 검색 — 전면 재검토

### 1-1. 🔴 "도톤보리 맛집" 검색 시 2개만 나옴 (번화가인데도)

**문제**: 도톤보리처럼 유명한 번화가를 검색해도 결과가 2개. 잘 모르는 동네는
아예 0개. 실시간(Google) 검색 결과 섹션이 배포 환경에서 나타나지 않음.

**원인 (이중 구조)**:
1. **큐레이션(수제 목업) 데이터의 한계** — `discoverData.ts`에 "도톤보리" +
   음식점 태그로 매칭되는 스팟이 딱 2개(타코야키 왕골목, 오코노미야키 미즈노).
   수제 데이터는 아무리 늘려도 "모든 동네"를 커버할 수 없음. 구조적 한계.
2. **실시간 검색이 배포에서 죽어 있음** — 코드상 실시간 Google Places 검색
   섹션은 이미 구현돼 있으나(`fetchLivePlaceSearch` → `/api/places/search`),
   서버가 mock으로 fallback하면 섹션 자체를 숨기게 되어 있음. 배포에서 안
   보인다 = **서버의 Google API 호출이 실패하고 있다**는 뜻.
   가장 유력한 실패 원인은 아래 1-2(키 미분리).

**수정 방안**:
- [ ] 1-2의 키 분리 먼저 해결 (이것만으로 도톤보리 검색이 실제 구글 결과
      최대 20개로 바뀔 것으로 예상)
- [ ] `/api/places/search` 실패 시 응답에 실패 사유 코드를 포함시키고
      (`sourceError: "referer_restricted" | "api_disabled" | ...`),
      Vercel 함수 로그로 확인 가능하게 임시 진단 로그 추가 → 원인 확정 후 제거
- [ ] **실시간 검색을 주(主) 결과로 승격** — 현재는 큐레이션이 메인이고
      실시간이 보조 섹션. 순서를 뒤집어 실시간 결과가 카테고리 칩·페이지네이션의
      대상이 되고, 큐레이션은 "추천 코스/에디터 픽" 보조 섹션으로 강등
- [ ] Google `searchText`의 `pageToken` 기반 실제 페이지네이션 구현
      (현재 최대 20개 1회 호출 — "수십 개" 기대치를 채우려면 페이징 필수)
- [ ] Kakao 국내 검색도 `page` 파라미터로 페이징 (현재 1페이지 15개 고정)

### 1-2. 🔴 Google API 키 미분리 (실시간 검색 죽은 원인 + 보안 문제)

**문제**: Vercel에 `NEXT_PUBLIC_GOOGLE_MAPS_API_KEY`와 `GOOGLE_PLACES_API_KEY`
변수는 둘 다 있지만 **같은 키 값**을 넣어둠 (사용자 확인).

**원인/영향**:
- 클라이언트 키는 브라우저에 노출되므로 Google 콘솔에서 **HTTP referrer
  제한**을 걸어야 안전한데, referrer 제한이 걸린 키는 **서버에서 호출하면
  구글이 403으로 거부**함 → 서버 Places 검색(실시간 검색)이 통째로 실패
- 반대로 referrer 제한을 안 걸면 키가 노출된 채 무제한 → 도용 시 과금 폭탄
- 즉 한 키로는 두 용도를 동시에 안전하게 만족시킬 수 없음

**수정 방안** (Google Cloud 콘솔에서, 코드 수정 불필요):
- [ ] 키 2개로 분리 생성:
  - **클라이언트 키** (`NEXT_PUBLIC_GOOGLE_MAPS_API_KEY`):
    API 제한 = Maps JavaScript API (+ Places API New — 플래너 자동완성용),
    애플리케이션 제한 = HTTP referrer (`travel-xi-red.vercel.app/*`,
    `*.vercel.app/*` 프리뷰 포함 여부는 선택)
  - **서버 키** (`GOOGLE_PLACES_API_KEY`):
    API 제한 = Places API (New)만, 애플리케이션 제한 = 없음(또는 가능하면
    IP — Vercel은 고정 IP가 아니므로 API 제한만으로 운용)
- [ ] 두 키 모두 **Places API (New)** 가 프로젝트에서 활성화됐는지 확인
      (구 Places API와 별개 항목임 — 여기서 자주 걸림)
- [ ] Vercel 환경변수 교체 후 재배포 → "도톤보리 맛집"으로 실검증

### 1-3. 🔴 큐레이션 스팟의 좌표가 가짜 (지터 생성값)

**문제**: 탐색 탭의 생성 스팟들은 실좌표가 아니라 도시 중심점 ± 랜덤 오프셋.
이 카드를 일정에 담으면 존재하지 않는 위치에 핀이 찍히고, 사용자가 그 위치를
믿고 이동할 수 있음. 가짜 평점(이미 제거)보다 실사용 위험이 큼.

**수정 방안**:
- [ ] 1-1의 "실시간 검색 주 결과 승격"이 근본 해결 (실좌표만 노출됨)
- [ ] 과도기: 큐레이션 스팟을 일정에 담는 순간 이름으로 Places API 1회
      resolve해서 실좌표로 교체, 실패 시 "위치 확인 필요" 뱃지 표시
- [ ] 손으로 넣은 유명 명소(글리코 사인 등 실좌표 입력분)는 유지 가능

### 1-4. 🟡 검색 UX 세부 불편

- **잘 모르는 동네 검색** — 큐레이션에 없으면 "결과가 없어요"로 끝. 실시간
  검색 살린 뒤에도, 0건일 때 "근처 지역 ○○ 결과 보기" 같은 대안 제시 필요
- **검색 결과에 사진이 없음** — 그라데이션 카드뿐. Places Photo API로 실제
  장소 사진 1장씩 (필드마스크에 `places.photos` 추가, 프록시 라우트로 서빙)
- **실시간 섹션 로딩 스켈레톤 없음** — 큐레이션은 스켈레톤이 있는데 실시간
  섹션은 로딩 표시 없이 늦게 나타남 → 스켈레톤 통일
- **최근 검색어가 /discover에만 있음** — /planner 검색창에도 적용
- **자동완성 부재(/discover)** — /planner에는 Google 자동완성이 있는데
  /discover 메인 검색창에는 없음. 동일 컴포넌트 재사용 검토

## 2. 데이터 유실 — 실사용 최우선 수정

### 2-1. 🔴 새로고침하면 일정이 전부 사라짐

**문제**: 일정을 다 짜고 브라우저를 닫으면 소멸. 실서비스라면 첫날 이탈 사유.

**원인**: `itineraryStore`의 persist가 `savedPlaces`만 저장
(`partialize: (state) => ({ savedPlaces: state.savedPlaces })`).
일정(`items`), 지도 장소(`places`), `activeDate`, `currentCity` 전부 휘발.
DB 저장은 로그인 + 수동 저장 버튼을 눌러야만 동작.

**수정 방안**:
- [ ] persist 대상에 `items`, `places`, `activeDate`, `currentCity`, `region`
      추가 (localStorage 기준 즉시 해결, 마이그레이션 버전 넘버 올릴 것)
- [ ] 로그인 사용자는 변경 시 디바운스 자동 저장(기존 공유 push 로직 재활용)
- [ ] "저장됨/저장 안 됨" 상태 인디케이터를 헤더에 표시

### 2-2. 🔴 헤더가 자꾸 "Fukuoka · Yufuin"으로 돌아옴

**원인**: `currentCity` 기본값이 하드코딩 잔재 + 2-1 때문에 영속 안 됨.
추가로 `/planner` 진입 시 목업 후쿠오카 트렌드 핀이 무조건 지도에 깔려서
어느 도시를 계획하든 "후쿠오카 앱"처럼 보임.

**수정 방안**:
- [ ] 기본값을 중립 문구("새 여행")로 교체
- [ ] `currentCity`를 persist에 포함 (2-1과 함께)
- [ ] TrendSheet의 후쿠오카 목업 핀 자동 주입 제거 — 시트를 열었을 때만,
      그리고 현재 도시 기준 데이터로 로드 (장기적으로는 실 API 트렌드)

## 3. 화면감 · 이용 편의 (UX 감사)

### 3-1. 🟡 화면감

- **다크모드 미지원인데 강제 다크 변환됨** — 삼성 인터넷 등의 강제 다크가
  라이트 전용 팔레트를 임의 반전시켜 스크린샷처럼 어둡고 깨진 화면이 됨.
  → `<meta name="color-scheme" content="light">` + CSS `color-scheme: light`
  선언으로 강제 변환 차단 (정식 다크모드는 별도 과제)
- **장소 사진 부재** — 카드 전부 그라데이션. 실서비스 감각은 사진이 절반
  (1-4 Photo API 항목과 동일)
- **카톡 공유 미리보기가 영어** — `layout.tsx` metadata가
  "Travel Scheduler / Plan your trip...". 한국어 title/description +
  og:image + 파비콘 세트 필요
- **타임라인이 00:00부터 시작** — 모바일 첫 화면이 빈 새벽 슬롯으로 가득.
  → 기본 스크롤 위치를 08~09시로, 심야 구간은 접기(collapse)

### 3-2. 🟡 이용 불편

- **모바일 3일 컬럼이 비좁음** — 420px 폭에 3열 + 시간축. 모바일은 1일
  뷰 기본 + 좌우 스와이프로 날짜 이동, 3일 뷰는 태블릿 이상에서
- **일정 담기 흐름이 김** — 카드 → 모달(날짜) → 모달(시간) 매번 반복.
  → "빠른 담기"(다음 빈 슬롯 자동 배정) 기본 + 상세 배정은 옵션으로
- **온보딩/빈 상태 안내 부재** — 첫 진입 시 뭘 해야 할지 안내가 없음.
  빈 일정 상태에 "탐색에서 장소를 찾아 담아보세요 →" CTA 배치
- **하단 탭바 부재** — 탐색/계획/보관함 이동이 햄버거 메뉴 안에만 있음.
  모바일 앱 감각이라면 하단 고정 탭바가 표준
- **되돌리기 없음** — 일정 삭제·이동 실수 시 복구 불가. 삭제 토스트에
  "실행 취소" 버튼

### 3-3. 🟡 이용에 더 필요한 것 (실용화 필수 기능 후보)

- **국내 지도 카카오 렌더링** — 국내 여행 실용화 시 사실상 필수
  (현재 국내도 구글맵 — 국내 POI/도보 품질 열세)
- **PWA** — manifest + 아이콘 → 홈화면 추가 시 앱처럼 동작
- **일정 내보내기/공유 강화** — 이미지 저장(캡처용 뷰), 카카오톡 공유 템플릿
- **에러 모니터링 + 애널리틱스** — Sentry(또는 Vercel 자체) + 간단 지표.
  어디서 이탈하는지 몰라선 개선 불가

## 4. 운영·보안·비용

- 🟠 **`/api/places/search` 무방비 공개 프록시** — 인증/rate limit 없음.
  외부에서 긁어가면 내 구글 쿼터로 과금.
  → 최소: IP 기준 rate limit (Vercel KV or upstash), Origin 헤더 검증
- 🟠 **API 라우트 디버그 로그 잔재** — 구글/카카오 응답 전문을
  `JSON.stringify`로 콘솔 출력(14곳). 로그 비용·노이즈.
  → 키 분리 검증(1-2) 끝난 뒤 일괄 제거
- 🟠 **내부용 라우트 노출** — `/dev/map-test`, 구데모 `/share/[id]`.
  → 프로덕션에서 404 처리(env 가드) 또는 삭제
- 🟡 **DB/Auth 실검증 필요** — `DATABASE_URL`/`AUTH_SECRET`은 Vercel에
  있음(확인됨). 실제 소셜 로그인 → 저장 → 공유 링크 열기 전 구간을
  프로덕션에서 1회 통과 테스트
